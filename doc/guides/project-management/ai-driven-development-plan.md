# AI驱动开发总计划

## 1. 文档目的

本计划用于把《Immortality》从“策划与技术方案集合”推进到“可由多个 AI 执行器协同开发的游戏项目”。

目标不是让单个 agent 一次性开发完整 MMO，而是先建立：

- 统一的仓库真相源
- 统一的任务拆分方法
- 统一的设计确认流程
- 统一的执行与校验协议
- 分阶段推进的全局路线图

## 2. 项目定位

### 2.1 产品定位

- 类型：修仙题材 MMORPG
- 当前阶段：MVP / Alpha 前置研发
- 首要验证卖点：真实时间修炼、基础天道规则、最小战斗循环
- 研发方式：`你负责方向和审批 + AI 执行器负责实现 + 文档负责约束`

### 2.2 当前仓库定位

- `doc/`：策划、规则、架构、API、任务与设计结论
- `game/`：团结引擎客户端工程
- `server/`：.NET 8 服务端
- `deploy/`：容器化、监控、部署
- `.trae/rules/`：执行器必须遵守的仓库规则

## 3. 协作模式

### 3.1 角色划分

| 角色 | 职责 |
|------|------|
| 你 | 确认方向、审批设计、验收里程碑 |
| 当前助手 / TRAE Work | 负责总计划、任务拆分、上下文治理、文档维护 |
| TRAE IDE | 负责仓库内实际开发、修改、调试、验证 |
| Codex / Claude Code | 负责特定任务实现、独立审查、测试修复、并行研究 |

### 3.2 执行原则

1. 先更新文档与契约，再改代码。
2. 先做最小闭环，再扩内容和复杂系统。
3. 先做可验证的模块，再做抽象和优化。
4. 任何需要方向选择的事项，必须沉淀到“设计确认队列”。
5. 任何执行器都不得绕过 `doc/` 和 `.trae/rules/` 自行发明规则。

## 4. 总体阶段路线

## 阶段 P0：项目治理与环境定型

### 目标

建立统一仓库结构、开发流程、规则系统和最小工程可编译状态。

### 交付物

- `game/`、`server/`、`deploy/`、`doc/` 结构稳定
- `.trae/rules/` 生效
- 团结引擎工程初始化完成
- `.NET 8 SDK`、本地构建、Docker 基础验证完成
- 项目任务板、执行协议、设计确认队列建立

### 退出标准

- 服务端可本地构建
- `game/` 可被团结引擎正常打开
- 执行器可按任务模板领取工作

## 阶段 P1：MVP 核心纵切

### 目标

打通第一个真正可玩的闭环：

`注册/登录 -> 创建角色 -> 获取角色 -> 修炼进度 -> 基础战斗 -> 基础掉落 -> 返回角色状态`

### 交付物

- 认证系统真实可用
- 角色创建与角色读取可用
- 修炼系统基础闭环
- 单场景基础战斗
- 最小背包与掉落
- 客户端主界面与基础导航

### 退出标准

- 新账号可从零创建角色并进入游戏主循环
- 玩家可以完成至少一次修炼与至少一次基础战斗
- 战斗与修炼结果可正确回写到玩家状态

## 阶段 P2：MVP 打磨与内容补足

### 目标

把“技术打通”升级为“可以体验并评审”的内部版本。

### 交付物

- 新手引导
- UI 风格统一
- 数值第一轮平衡
- 第一批怪物、掉落、装备、任务内容
- 数据导入与配置校验工具
- 基础埋点与日志面板

### 退出标准

- 连续游玩 30-60 分钟有清晰成长体验
- 核心循环不依赖开发者手工干预
- 数值曲线可被调整且能复现

## 阶段 P3：Alpha 前验证

### 目标

验证多人、压力、观测和内容节奏。

### 交付物

- 基础并发测试
- 关键链路监控
- 数据恢复与错误排查工具
- 运营后台基础能力
- Alpha 测试说明与玩家招募准备

### 退出标准

- 服务端能在预期测试规模下稳定运行
- 关键指标可观测
- 已建立问题反馈和修复流程

## 阶段 P4：Alpha 扩展与 Beta 准备

### 目标

补齐多人内容、社交与中期成长玩法。

### 交付物

- 宗门 / 社交基础
- 多副本与更多世界区域
- 更完整的装备、炼丹、功法内容
- 更严谨的经济系统和反滥用机制

### 退出标准

- 中期内容不再只依赖单一修炼/战斗循环
- 多人互动能形成真实测试价值

## 5. 工作流分层

### 5.1 治理层

- 仓库结构规则
- API 契约
- 任务模板
- 设计确认队列
- 任务总板

### 5.2 执行层

- `game/` 客户端开发
- `server/` 服务端开发
- `deploy/` 运维与监控
- `tools/` 导表、校验、辅助脚本（后续可增加）

### 5.3 验证层

- 编译验证
- 自动化测试
- 文档一致性检查
- API 契约一致性检查
- 关键手工体验验证

## 6. 必须由你确认的设计域

下面这些不能交给执行器自行定夺，必须列入“设计确认队列”：

- 美术风格与参考图
- UI 视觉语言
- 主摄像机和操作手感
- 时间倍率与调试倍率
- 修炼节奏与突破惩罚
- 战斗节奏与操作复杂度
- 新手期目标和时长
- 世界范围与首批地图规模
- 音频风格与制作方式
- 商业边界、合规边界、IP 风险处理策略

## 7. 不在当前阶段优先做的事项

- 复杂社交和宗门战争
- 完整拍卖行和跨服经济
- 过早拆分大量微服务
- 大规模程序化内容生成
- 过早做移动端与多平台适配
- 为未来正式服过度优化架构

## 8. 当前推荐的实际推进顺序

1. 完成 P0 剩余收尾：SDK、团结工程初始化、构建验证。
2. 进入 P1：角色创建与角色读取。
3. 继续 P1：修炼进度读取、修炼推进、突破。
4. 继续 P1：基础战斗回路、最小背包与掉落。
5. 进入 P2：UI 风格统一、新手引导、第一轮平衡。

## 9. 与执行器配合的方法

每个执行器开始任务前必须读取：

1. `doc/guides/project-management/task-board.md`
2. 对应任务文档
3. `.trae/rules/*.md`
4. 相关 API / 架构 / 系统文档

每个执行器完成任务后必须输出：

- 修改了哪些文件
- 完成了哪些验收标准
- 还剩哪些未做项
- 有哪些阻塞需要你或总控决策

## 10. 成功标准

如果这套计划执行顺利，最终会形成这样的状态：

- 你可以在 `doc/` 中总览项目进度
- 任何执行器都能从文档中领取任务并获得足够上下文
- 重要设计决策不会只散落在聊天中
- 仓库不会因多 agent 并行而失控
- MVP 会沿着清晰的阶段性目标持续推进
