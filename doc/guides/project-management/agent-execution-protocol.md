# Agent执行协议

## 1. 文档目的

本协议用于约束 `TRAE IDE`、`Codex`、`Claude Code` 等执行器在本仓库中的工作方式，避免出现：

- 接口各写各的
- 目录结构漂移
- 文档和代码脱节
- 一次性改动过大难以验收

## 2. 执行器启动前必读顺序

每个执行器领取任务前，必须按顺序读取：

1. `doc/guides/project-management/task-board.md`
2. 当前具体任务文档或任务描述
3. `.trae/rules/01-repo-structure.md`
4. `.trae/rules/02-api-contracts.md`
5. `.trae/rules/03-game-client.md`
6. 与任务相关的系统文档、架构文档、API 文档

## 3. 任务认领规则

### 可直接执行的任务

- 目标明确
- 输入文档明确
- 验收标准明确
- 不需要你确认方向

### 必须暂停并请求确认的任务

- 美术风格
- UI 风格
- 战斗操作模式
- 时间倍率和成长节奏
- 突破惩罚
- 地图范围
- 商业/合规边界

这类任务必须先写入 [设计确认队列](./design-review-queue.md)。

## 4. 任务颗粒度

每个任务应满足以下原则：

- 1 次实现只解决 1 个清晰目标
- 理想工作量控制在 0.5 - 2 天
- 改动文件范围尽量集中
- 改动后能通过编译、测试或手工验证

### 好任务示例

- 打通登录后检查角色是否存在
- 完成角色创建接口与客户端联调
- 完成修炼进度查询与客户端展示

### 坏任务示例

- 完成客户端全部功能
- 做完整修炼 MMO 系统
- 把所有 UI 一次性重写

## 5. 交付格式

执行器完成任务后，必须明确交付：

- 修改了哪些文件
- 完成了哪些验收项
- 如何验证
- 还剩哪些未完成项
- 是否需要后续任务
- 是否引入了新依赖或新配置

## 6. 定义完成（Definition of Done）

一个任务只有在同时满足以下条件时，才可以标记为完成：

1. 代码或文档改动已经落库
2. 验收标准已逐条满足
3. 相关文档已同步
4. 能提供编译、测试或手工验证依据
5. 未遗留“伪完成”的占位逻辑

## 7. 文档同步规则

- 结构变更：先更新结构文档
- 接口变更：先更新 API 契约或接口文档
- 任务完成：更新任务总板
- 方向决策：更新设计确认队列

## 8. 校验规则

### 服务端任务最少校验

- 编译通过
- 关键接口可调用
- 错误响应明确
- 日志可定位问题

### 客户端任务最少校验

- 团结引擎工程可打开
- 目标场景/页面可进入
- 关键交互能完成
- 错误态有提示

### 配置或数据任务最少校验

- ID 唯一
- 引用不丢失
- 枚举值匹配
- 文档和资产一致

## 9. 多执行器协作模式

推荐采用以下角色分工：

- `执行器 A`：主实现
- `执行器 B`：独立 review
- `执行器 C`：测试或文档校对

### 推荐用法

- `TRAE IDE`：主实现与仓库内调试
- `Codex`：局部实现、数据处理、重构
- `Claude Code`：独立审查、复杂逻辑复核、并行研究

## 10. 禁止事项

- 不要在未确认的设计域擅自拍板
- 不要同时修改两套客户端实现
- 不要绕过 Gateway 直接在客户端硬编码内部服务地址
- 不要只改代码不改文档
- 不要把“TODO 占位实现”当作任务完成
