# Agent执行协议

## 1. 文档目的

本协议用于约束 `TRAE IDE`、`Codex`、`Claude Code` 等执行器在本仓库中的工作方式，避免出现：

- 接口各写各的
- 目录结构漂移
- 文档和代码脱节
- 一次性改动过大难以验收

## 2. 执行器启动前必读顺序

每个执行器领取任务前，必须按顺序读取：

1. `doc/guides/technical/global-constraints.md` — 全局硬约束速查
2. `doc/guides/project-management/task-board.md` — 任务总板
3. `doc/guides/project-management/current-task-packs.md` — 当前可领取任务包
4. 当前具体任务文档或任务描述
5. `.trae/rules/01-repo-structure.md` 至 `04-task-execution.md`
6. 与任务相关的系统文档（`doc/core-systems/`）、规则规范（`doc/specs/`）、架构文档
7. 若涉及资产，读取 `doc/guides/technical/asset-requirements-p1.md`

## 3. 任务认领规则

### 3.1 任务就绪标准（Definition of Ready）

一个任务只有满足以下全部条件，才可被认领：

1. 任务 ID 在 `task-board.md` 中存在且状态为 `[ ]` 或 `[~]`
2. 在 `current-task-packs.md` 中有详细任务包，或在任务描述中补齐了目标/范围/验收标准
3. 前置任务已完成（`[x]`）或明确标注为可并行
4. 不依赖未确认的设计决策（若依赖，需已在 `design-review-queue.md` 中确认）
5. 涉及的环境前置（SDK、编辑器、数据库）已就绪或任务本身即环境前置

### 3.2 可直接执行的任务

- 目标明确
- 输入文档明确
- 验收标准明确
- 不需要你确认方向

### 3.3 必须暂停并请求确认的任务

- 美术风格
- UI 风格
- 战斗操作模式
- 时间倍率和成长节奏
- 突破惩罚
- 地图范围
- 商业/合规边界

这类任务必须先写入 [设计确认队列](./design-review-queue.md)。

## 4. 任务颗粒度

每个任务应满足以下原则：

- 1 次实现只解决 1 个清晰目标
- 理想工作量控制在 0.5 - 2 天
- 改动文件范围尽量集中
- 改动后能通过编译、测试或手工验证

### 好任务示例

- 打通登录后检查角色是否存在
- 完成角色创建接口与客户端联调
- 完成修炼进度查询与客户端展示

### 坏任务示例

- 完成客户端全部功能
- 做完整修炼 MMO 系统
- 把所有 UI 一次性重写

## 5. 执行流程

每个任务按以下闭环执行：

```
认领 → 读上下文 → 发现真实代码 → 实现 → 编译门禁 → 验证 → Git 检查点 → 回写状态 → 交接
```

### 5.1 发现真实代码

实现前，必须搜索和阅读任务包"输入上下文"中列出的文件，确认：

- 文件实际存在且内容与文档描述一致
- 涉及的方法、类、字段在代码中可找到
- 若文档与代码不一致，以代码为准并在交付时标注差异

### 5.2 资产获取优先级

需要视觉/音频资产时，按以下优先级获取，不得跳级：

| 优先级 | 策略 | 说明 |
|--------|------|------|
| **P0** | 复用现有资产 | 在 `game/Assets/` 中搜索可复用的模型、精灵、音频、材质 |
| **P1** | 用 Unity 原语占位 | Cube/Sphere/Capsule/Plane + 纯色材质，不阻塞功能开发 |
| **P2** | AI 生成 | 调用 TJGenerators 生成，需美术风格已确认 |
| **P3** | 外部导入 | 从资产库搜索下载或手动导入 |

> P1 阶段默认使用 **P1（Unity 原语占位）**，除非任务包明确要求生成资产。

### 5.3 编译门禁

实现完成后，必须通过编译门禁才可进入验证阶段。

**服务端门禁流程：**

```
cd server
dotnet build
```

- 编译必须 0 error（warning 可接受但需记录）
- 若有 error → 修复 → 重新编译 → 直到 0 error
- 最多 3 轮修复，仍失败则在交付中记录阻塞原因

**客户端门禁流程：**

客户端编译依赖团结编辑器。若编辑器可用：

1. 切换 Console 窗口焦点
2. Clear Console
3. 等待编译完成
4. 读取 Console 输出
5. 确认 0 error

若编辑器不可用（P0-ENV-002 未完成），在交付中标注"客户端编译待验证"。

### 5.4 Git 检查点

每完成一个任务（或一个独立子任务），必须创建 Git 提交：

```
git add <本次改动文件>
git commit -m "<type>(<scope>): <简要描述> [task-id]"
```

**提交类型：**

| type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 |
| `refactor` | 重构（不改行为） |
| `docs` | 文档 |
| `chore` | 构建/配置/依赖 |
| `test` | 测试 |

**示例：**

```
git commit -m "feat(auth): 登录后检查角色状态闭环 [P1-CHAR-001]"
git commit -m "fix(server): 修正 Gateway 缺失 Yarp 包引用 [P0-ARCH-005]"
git commit -m "docs(pm): 补充资产需求清单 [P0-PM-006]"
```

**规则：**

- 一个提交只包含一个任务范围的改动
- 不要把无关文件混入提交
- 不要在一个提交中混合多个任务
- 提交前确认 `.gitignore` 已排除缓存和构建产物

## 6. 交付格式

执行器完成任务后，必须明确交付：

- 修改了哪些文件
- 完成了哪些验收项
- 如何验证（编译结果、测试结果、手工验证步骤）
- 还剩哪些未完成项
- 是否需要后续任务
- 是否引入了新依赖或新配置
- Git 提交 hash

## 7. 定义完成（Definition of Done）

一个任务只有在同时满足以下条件时，才可以标记为完成 `[x]`：

1. 代码或文档改动已通过 Git 提交落库
2. 验收标准已逐条满足
3. 相关文档已同步
4. 已通过编译门禁（0 error）或标注环境阻塞
5. 未遗留"伪完成"的占位逻辑
6. 任务总板状态已回写

> 若仅完成代码编写但未通过编译门禁或验证，标记为 `[~]`（已实现，待验证）。

## 8. 文档同步规则

- 结构变更：先更新结构文档
- 接口变更：先更新 API 契约或接口文档
- 任务完成：更新任务总板
- 方向决策：更新设计确认队列
- 新增约束：更新 `global-constraints.md`

## 9. 校验规则

### 服务端任务最少校验

- `dotnet build` 0 error
- 关键接口可调用（curl 或集成测试）
- 错误响应明确（有明确 HTTP 状态码和 message）
- 日志可定位问题（Serilog 结构化日志）

### 客户端任务最少校验

- 团结引擎工程可打开
- 目标场景/页面可进入
- 关键交互能完成
- 错误态有提示
- Console 0 error

### 配置或数据任务最少校验

- ID 唯一
- 引用不丢失
- 枚举值匹配
- 文档和资产一致

## 10. 多执行器协作模式

推荐采用以下角色分工：

- `执行器 A`：主实现
- `执行器 B`：独立 review
- `执行器 C`：测试或文档校对

### 推荐用法

- `TRAE IDE`：主实现与仓库内调试
- `Codex`：局部实现、数据处理、重构
- `Claude Code`：独立审查、复杂逻辑复核、并行研究

### 执行器交接格式

当一个执行器完成任务后需要交接给下一个执行器时，在任务回写中使用以下格式：

```markdown
### 交接记录

**任务 ID**: P1-CHAR-001
**执行器**: TRAE IDE
**状态**: 已实现，待验证 [~]

**已完成**:
- 登录后调用 /api/player/me
- 未创建角色时进入创角分支
- 已有角色时进入主界面

**未完成 / 阻塞**:
- 需 P0-ENV-001 完成后验证 dotnet build
- 需 P0-ENV-002 完成后验证客户端编译

**Git 提交**: a1b2c3d
**交接给**: Claude Code（审查接口与状态流）
```

## 11. 禁止事项

- 不要在未确认的设计域擅自拍板
- 不要同时修改两套客户端实现
- 不要绕过 Gateway 直接在客户端硬编码内部服务地址
- 不要只改代码不改文档
- 不要把"TODO 占位实现"当作任务完成
- 不要跳过编译门禁直接标记任务完成
- 不要在一个 Git 提交中混合多个任务的改动
- 不要删除 `.gitkeep` 文件（它们保持目录结构）
