# 阶段审查清单

> 本清单用于在“开始执行前 / 准备标记完成前 / 准备提交前”进行统一自查，减少任务板、代码和真实状态不一致的问题。

## 任务可执行性检查

- [ ] 任务目标、输入上下文、范围和验收标准完整
- [ ] 前置任务、阻塞条件和人工前提已写清
- [ ] 涉及本机安装、编辑器操作、账号授权的任务已标注为“人工主导，AI 辅助”
- [ ] 任务包没有默认依赖不存在的方法、文件或目录

## 代码闭环检查

- [ ] 服务端新增调用已补齐缺失的 `using`、依赖包和配置项
- [ ] 客户端任务若依赖 `ApiClient`、Manager 或 DTO，已明确是否需要新增方法或映射
- [ ] 登录后、无角色、未认证、接口失败等关键分支都有明确状态流
- [ ] 文档描述与当前代码实现一致，没有“任务写得能做但代码入口不存在”的情况

## 数据与运行检查

- [ ] 数据库基础表是否已有初始迁移或明确初始化步骤
- [ ] 若任务依赖 PostgreSQL / Redis / Gateway / 团结编辑器，已说明最小验证方式
- [ ] 无法在当前机器完成的验证，已明确记录为环境阻塞，而不是默认视作完成

## 状态回写检查

- [ ] 只有在"完成并验收"后才把任务标为 `[x]`
- [ ] 若仅完成代码编写但未通过编译门禁或验证，标为 `[~]`（已实现，待验证）
- [ ] 新发现的前置任务、阻塞项或设计确认项，已回写到任务总板或设计确认队列
- [ ] 交接记录已按 agent-execution-protocol.md 第 10 节格式写入

## 提交前检查

- [ ] `.gitignore` 已排除缓存、构建产物和本地代理日志
- [ ] 提交仅包含本次任务应有的改动，没有混入无关临时文件
- [ ] 如新增治理文档，已接入导航或在总览页可达
- [ ] Git 提交消息格式：`<type>(<scope>): <描述> [task-id]`
- [ ] 一个提交不包含多个任务的改动
